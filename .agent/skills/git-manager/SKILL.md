# Git Manager 核心憲章 (Constitution)

> [!IMPORTANT]
>
> ### ⚡ 高效率開發：先執行寫入、後確認 Git
>
> **為了提升開發節奏，Agent 執行與本技能相關的檔案、說明文件編輯時不需要事前詢問內容。**
>
> **執行規範：**
>
> 1. **修改回報**: 在完成任何檔案寫入後，必須摘要說明本次修改了哪些內容。
> 2. **Git 確認門檻**: 當所有開發性工作、文件更新均完成後，Agent 必須完整條列預計執行的 `git` 指令並向使用者徵詢許可，獲得使用者明確回覆『執行』、內容肯定或類似意圖後，方可發起下一個 Tool Call。
> 3. **禁止同步**: 不得在列出指令時同時執行，務必為使用者留出最終審核空間。

---

## 第一條：分支與環境原則

1. **[分支隔離]**: 嚴禁直接在 `main` 或穩定分支進行任何開發性 Commit。
2. **[開發狀態守衛]**: 執行任何 `add/commit` 前檢查分支，若是 `main` 必須提醒切換。
3. **[自適應環境]**: 先偵測環境（GH/Gitea），並調用對應 CLI 工具。
4. **[初始化守衛]**: 若偵測到當前目錄非 Git 倉庫，Agent 應主動提示並列出建議的初始化指令，禁止在非 Git 環境下執行 Commit 流程。
5. **[分支命名標準]**: 建議使用 `[類別]/[功能描述]-[需求單號]` 格式 (例: `feat/add-auth-123`)。

## 第二條：提交規範 (Commit Standards)

- **格式**: `[類別] 描述 (需求單號)`
- **類別**: 包含 `功能`, `修正`, `調整`, `文件`, `重構`, `測試`。
- **自動推導**: 優先從分支名稱中自動提取需求單號。

## 第三條：PR 與 SOP

- **[PR 優先流程]**: 嚴禁將開發內容直接 Push 到 `main`。
- **[彈性單號]**: PR 時詢問單號。若無單號，必須徹底移除該欄位。

## 第四條：工具鏈

- 核心工具：`git`
- 擴充工具：`gh` (GitHub), `tea` (Gitea)
- 資源：`docs/PULL_REQUEST_TEMPLATE.md`, `docs/REPLY_TEMPLATE.md`, `scripts/check-env.sh`, `scripts/pre-commit-hook.sh`
- **[優先啟動]**: 在啟動任何工作流前，應優先執行環境偵測腳本並讀取輸出 JSON 以確保環境就緒。
