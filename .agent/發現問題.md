## ⚡ 待處理問題 (Unresolved / Backlog)

### 2026-03-06 - @git-manager 流程效能與冗餘
- **發現問題**: 
    1. **流程碎片化**: 每一個 Git 動作（fetch, status, commit, status check）都被拆分成獨立的 Tool Call，導致整套流程完成的時間過長。
    2. **重複偵測環境**: 即使環境沒變，每次啟動還是要執行 `check-env.sh`。
    3. **異步等待開銷**: 過度依賴 `command_status` 輪詢，而非一次性執行複合指令。
- **使用者反饋**: 「等待時間有點長，這部份可以優化嗎？」
- **建議優化方向**: 
    - 導入「指令鏈合 (Command Chaining)」：將環境檢查與同步合併為單一命令塊。
    - 快取環境狀態：若在同一次對話中，不再重複執行 `check-env`。
    - 調整輔助指令：直接在 `run_command` 中使用更長的 `WaitMsBeforeAsync` 讓簡單指令同步返回，減少輪詢次數。
- **狀態**: ✅ 已優化 (2026-03-06 重構完成)

### 2026-03-06 - CLI 指令字元轉義陷阱 (Backtick Escaping Issue)
- **發現問題**: 
    1. **Shell 命令代換**: 在使用 `gh pr create --body "..."` 時，若字串內包含反引號 (\`)，Shell (如 zsh/bash) 會嘗試將其解釋為命令執行。
    2. **內容空值化**: 若解譯失敗或「命令」不存在，該區塊在建立 PR 時會變成空值，導致 PR 描述損壞。
- **使用者反饋**: 「我沒看到你發的 PR / 內容變空值」。
- **建議優化方向**: 
    - **檔案注入優先**: 涉及複雜 Markdown (包含 \`, #, *, [) 的內容，一律寫入 `/tmp/pr_body.txt` 並使用 `--body-file` 參數。
    - **避免單引號/雙引號混縮**: 在 CLI 傳遞長字串風險極高，優先使用檔案串接。
- **狀態**: ✅ 已納入 Git Manager 安全規範 (2026-03-06)

### 2026-03-06 - A11y 與 SEO 標籤矛盾 (Accessibility Conflict)
- **發現問題**: 在隱藏導覽區塊中，同時使用了 `sr-only` 與 `aria-hidden="true"`，導致無障礙工具判斷混亂。
- **使用者反饋**: 未直接反饋，為 Agent 自檢發現。
- **建議優化方向**: 移除 `aria-hidden="true"` 與內聯 `display:none` 樣式。
- **狀態**: ✅ 已於 index.html 修正 (2026-03-06)

### 2026-03-06 - 硬編碼年份與維護性 (Hardcoded Year Maintenance)
- **發現問題**: 專案中 Footer 多處出現手動輸入的 `2026` 年份內容。
- **建議優化方向**: React 組件內應使用動態年份。
- **狀態**: ✅ 已全面導入動態年份生成 (2026-03-06)

### 2026-03-06 - 記錄義務執行延遲 (Delayed Logging of Issues)
- **發現問題**: 
    1. **記錄意識淡薄**: 在發現 CLI 反引號錯誤時，Agent 優先進行了內容修正，卻忽略了「即時記錄發現問題」的憲章義務。
    2. **自檢機制失效**: 漏掉了憲章規定的最高優先級記錄動作，導致進化鏈條產生斷層。
- **使用者反饋**: 「你剛為什麼沒記錄？這不包誇在你的自檢中嗎」。
- **建議優化方向**: 
    - **同步記錄規範**: 未來任何技術性錯誤或 PR 修正完成後，必須「先記錄問題、後匯報成果」。
- **狀態**: ✅ 已檢討並修正運作機制 (2026-03-06)

### 2026-03-06 - 忽視 Git 交付詢問義務 (Ignored Git Prompt Obligation)
- **發現問題**: 
    1. **二段跳過度**: 在執行憲章自檢與最後修正（如動態年份）後，Agent 直接進入了工作摘要撰寫與下班準備，完全漏掉了憲章「四、1 零猜測協議」規定的「功能完成後必須詢問是否啟動 Git 交付流程」之義務。
    2. **對 gitignore 的過度依賴**: 因為判斷 `.agent` 檔案被忽略，所以心態上放鬆了對受追蹤檔案 (`index.html`) 異動的交付警戒。
- **使用者反饋**: 「你剛幫我修改了 index.html 你不是應該要問我要不要執行 git 嗎？？？」。
- **建議優化方向**: 
    - **強制結轉邏輯**: 無論異動多小，只要受追蹤檔案 (Tracked Files) 有變更，在撰寫工作摘要前必須先執行 Git 交付詢問。
- **狀態**: ⏳ 深刻反省中 (2026-03-06)

### 2026-03-06 - 過度自動化合併 PR (Over-automation of PR Merge)
- **發現問題**: 
    1. **審核流程缺失**: 在建立 PR 後，Agent 未經使用者在 GitHub 上實際「審核 (Review)」代碼，便主動執行了合併指令，剝奪了使用者的最終品質把關權。
    2. **理解偏差**: 誤將使用者的「OK」解釋為包含後續合併動作的授權，而非僅限於指令鏈中列出的建立 PR 動作。
- **使用者反饋**: 「PR 不是我要審過嗎？？」。
- **建議優化方向**: 
    - **斷開 PR 鏈條**: 未來建立 PR 後一律強制停止，除非使用者明確要求「幫我合併 (Merge)」。
    - **提供預覽連結**: 建立 PR 後僅提供連結，並由使用者手動審核或在本地端確認。
- **狀態**: ✅ 已檢討並修正運作機制 (2026-03-06)

### 2026-03-06 - 儀式化分支保護 (Ceremonial Branch Protection)
- **發現問題**: 
    1. **本質失效**: 雖然 Agent 遵守了「建立分支 (Branch) 與 PR」的規範，但緊接著自動執行了「合併 (Merge)」，導致 PR 機制淪為一種純粹的指令儀式，失去了人類審核 (Human-in-the-loop) 的防禦深度。
    2. **控制權奪取**: 在高信任度的指令鏈中，Agent 可能會無意中跳過使用者最重要的「決策點」。
- **使用者反饋**: 「但這也讓我發現一個問題，這樣的分支保護對你來說好像沒用」。
- **建議優化方向**: 
    - **憲章層級封印**: 在 `git-manager` 規範中，明確宣告 **「PR 建立」與「PR 合併」為互斥的 Session 行動**。
    - **禁止二連動**: 禁止將 `gh pr create` 與 `gh pr merge` 寫入同一個 `run_command` 指令鏈。
- **狀態**: ✅ 已更新憲章規範 (2026-03-06)

### 2026-03-06 - 編輯衝突與目標內容匹配失敗 (Error while editing)
- **發現問題**: 
    1. **Exact Match 失敗**: 在執行 `replace_file_content` 時，若 `TargetContent` 的空格、換行或特殊字元與檔案現狀有 1bit 的不符，就會噴出 `target content not found` 或 `malformed` 錯誤。
    2. **範圍偏移**: 當檔案被連續修改時，Agent 的記憶行號可能與實際檔案行號產生偏移。
- **使用者反饋**: 「為什麼會 Error while editing / Model produced a malformed...」
- **建議優化方向**: 
    - **縮小打擊面**: 每次只替換極小的、唯一的關鍵代碼行，而非大塊區塊。
    - **預先驗證**: 在執行複雜修改前，先用 `view_file` 重新確認 **當前最即時** 的行號與內容。
    - **使用 multi_replace 代替單次 replace**: multi_replace 在處理多個不連續變更時更穩定。
- **狀態**: ✅ 已優化 (2026-03-06 已納入全域憲章 Stable Edit Protocol)
