# 工程師評價心理學：深度學習筆記

互動網站: [https://gemini.google.com/share/6a9569c51762](https://gemini.google.com/share/6a9569c51762)

這份筆記整理了工程師在職場中面對「評價」時的心理機制、常見行為模式，以及如何建立健康的自我認同體系。

---

## 一、核心問題：為什麼我們如此在意評價？

工程師的工作產出（代碼、架構）高度抽象且具備創造性。當產出受到批評時，大腦往往無法區分「對工作的建議」與「對個人的否定」。

### 心理學三大根源

- **生存本能（Survival Instinct）**  
  在演化過程中，被部落排斥意味著死亡。大腦邊緣系統會將「負面評價」視為物理威脅，觸發皮質醇分泌，導致心跳加速與焦慮。

- **鏡像自我（Looking-glass Self）**  
  查爾斯·庫利（Charles Cooley）提出。我們對自己的定義，其實是我們觀察到「別人如何看待我」的投影。評價成了我們確認自身能力的座標。

- **社會比較理論（Social Comparison Theory）**  
  利昂·費斯廷格（Leon Festinger）提出。在缺乏絕對標準時，人類傾向透過與同儕比較（如職級、薪資）來定義成功。

## 二、工程師日常生動案例分析

透過具體場景，識別這些心理機制如何影響開發者的行為：

| 案例角色 | 核心情境 | 心理分析 |
|----------|----------|----------|
| **老王的 PR 焦慮** | 收到 Request Changes 或「邏輯笨」的評論 | 外在計分卡主導。將代碼品質與智力掛鉤，導致防禦心態 |
| **小美的技術債恐懼** | 擔心改壞代碼影響「穩重」形象，故不敢重構 | 生存本能與社交安全性。為了維持零失誤評價而犧牲技術價值 |
| **阿強的技術焦慮** | 盲目引入大廠新框架，追求技術流行 | 社會比較理論。在跟「社群幻覺」比較，試圖透過工具換取認可 |
| **大力的救火英雄** | 享受半夜第一個修 Bug 的「被需要感」 | 效用認可沉迷。成功建立在被需要的頻率，而非長期產出影響力 |
| **曉芬的文檔完美主義** | 花費過多時間潤飾文檔，擔心一個錯字毀專業感 | 完美主義下的評價權放大。在邊際效益極低的工作上消耗過度 |

## 三、實踐解決方案：奪回評價主導權

### 1. 建立「內在計分卡」（Inner Scorecard）
華倫·巴菲特（Warren Buffett）提出的概念。重點在於你是用「自己的標準」還是「別力的標準」來衡量成功。

- **內在指標**：今天的我比三個月前更進步嗎？我解決了真實的問題嗎？  
- **外在指標**：GitHub Stars、主管的評分字母、名片上的頭銜。

### 2. 自我與代碼的解耦（Decoupling）
- **心態重構**：代碼離開編輯器後，它就是一個獨立的邏輯實體。  
- **定義**：被 Review 的是「代碼的時空複雜度」，而不是「你的人格」。

### 3. 擁抱「凡人化心態」
- 承認自己也有不懂的地方，這並不會損害資深工程師的威信。  
- 主動提問能建立團隊的「心理安全性」，讓溝通回到解決問題的本質。

## 四、總結與座右銘

> **「評價是一項工具，而不應是你的主人。」**

當你專注於內在成長時，外部的認可會作為「副產品」自然隨之而來，而不再是你焦慮的源頭。
